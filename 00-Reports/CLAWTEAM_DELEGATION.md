# Clawteam 委托策略

**核心原则：重复性任务 + 复杂拆解任务 → 自动委托 clawteam，主 Agent 只负责验收和监控。**

---

## 一、任务分类与路由

### 🔁 自动委托给 clawteam 的任务类型

| 类型 | 示例 | 触发条件 |
|------|------|----------|
| **数据获取失败** | API 超时、爬虫报错、数据源不可用 | 任何数据获取异常 |
| **模块报错** | DSS 模块崩溃、依赖缺失、配置错误 | 任何运行时错误 |
| **代码调试** | 找 bug、分析堆栈、修复逻辑 | 错误信息出现 |
| **批量操作** | 批量处理文件、批量转换、批量分析 | >3 个相似操作 |
| **研究任务** | 技术调研、方案对比、文档查阅 | 需要 web_search + 分析 |
| **代码重构** | 模块拆分、性能优化、代码清理 | 明确的重构目标 |
| **测试生成** | 单元测试、集成测试、E2E 测试 | 需要生成测试代码 |

### 🎯 主 Agent 保留的任务

| 类型 | 原因 |
|------|------|
| 用户对话/澄清 | 需要理解用户意图 |
| 敏感操作确认 | 发送邮件、推文、公开内容 |
| 最终验收 | 决定结果是否可接受 |
| 简单编辑 | 单文件小改动，直接用 edit |
| 文件读取 | 直接用 read，无需委托 |

## 二、委托流程

```
用户请求/异常触发
       ↓
  主 Agent 判断
       ↓
   ┌───┴───┐
   │ 委托? │
   └───┬───┘
       │
    是 │ 否 → 直接处理
       ↓
  sessions_spawn({
    runtime: "subagent",
    agentId: "clawteam",
    task: "具体任务描述",
    mode: "run"
  })
       ↓
  clawteam 执行（自动推送结果）
       ↓
  主 Agent 验收
       ↓
   ┌───┴───┐
   │ 通过? │
   └───┬───┘
       │
    是 │ 否 → 重新委托或手动修复
       ↓
  返回用户
```

## 三、验收标准

### ✅ 通过条件

1. **任务完成**：明确回答了问题或解决了问题
2. **无工具调用异常**：sessions_history 显示工具调用成功
3. **输出质量**：结果符合预期格式和内容
4. **无幻觉**：没有编造不存在的信息

### ❌ 重新委托条件

1. clawteam 报告"无法完成"
2. 工具调用失败率 > 50%
3. 输出格式不符合要求
4. 明显的逻辑错误

### 🔄 升级处理

如果同一任务委托 2 次仍失败：
1. 主 Agent 接手，手动诊断
2. 记录失败原因到 memory/
3. 更新 clawteam 的上下文或配置

## 四、监控机制

### 任务级监控

```javascript
// 委托后检查
subagents({ action: "list" })

// 检查点：
// - status: running → done
// - runtime: 合理的执行时间
// - model: 确认使用了正确模型
```

### 结果级监控

```javascript
// 获取执行历史
sessions_history({ sessionKey: "<child-session-key>" })

// 检查点：
// - tool calls 是否成功
// - 错误信息（如有）
// - 最终输出的质量
```

### 周期性健康检查

```javascript
// 每次 heartbeat 时可选执行
sessions_spawn({
  agentId: "clawteam",
  task: "健康检查：回复 OK",
  mode: "run"
})
```

## 六、记忆锚点

**决策流程：**
1. 判断 → 委托？ → spawn clawteam
2. 等待推送 → 验收
3. 通过？ → 返回用户 / 失败？ → 重试或接手

**验收检查：**
- `subagents list` → 状态监控
- `sessions_history` → 结果审计

**升级条件：**
- 同一任务 2 次失败 → 主 Agent 接手

## 七、示例场景

### 场景 1：DSS 数据获取失败

```
用户：DSS 今天数据获取失败了

主 Agent：
  1. 判断：数据获取失败 → 自动委托
  2. spawn clawteam: "DSS 数据获取失败，请诊断并修复。检查 eastmoney_crawler.py 和 news_crawler.py"
  3. 等待结果推送
  4. 验收：检查是否成功获取数据
  5. 返回用户："已修复，原因是 [原因]"
```

### 场景 2：批量文件处理

```
用户：把这 10 个 PDF 都总结一下

主 Agent：
  1. 判断：批量操作 (>3) → 自动委托
  2. spawn clawteam: "批量总结以下 PDF：[列表]"
  3. 等待结果推送
  4. 验收：检查所有 PDF 是否都被处理
  5. 返回用户：整合后的总结
```

### 场景 3：复杂代码调试

```
用户：这个函数有 bug，帮我找一下

主 Agent：
  1. 判断：代码调试 → 自动委托
  2. spawn clawteam: "调试 [文件:函数]，问题描述：[...]"
  3. 等待结果推送
  4. 验收：检查修复是否合理
  5. 返回用户："问题在于 [原因]，已修复"
```
